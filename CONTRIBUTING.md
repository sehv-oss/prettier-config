# Contributing

First off, thank you for taking the time to contribute! ❤️

All types of contributions are encouraged and valued, no matter how small. Whether it's reporting a bug, suggesting a feature, improving documentation, or writing code — every bit helps, and we appreciate it.

> And if you like the project but don't have time to contribute, that's fine too! You can also support it by starring the repository, sharing it, or mentioning it to others.

## Code of Conduct

This project and everyone participating in it is governed by our [Code of Conduct](CODE_OF_CONDUCT.md). By participating, you are expected to uphold this code.

## How Can I Contribute?

### Reporting Bugs

Before opening a bug report, please search the existing issues to see if the problem has already been reported. If it has, add a comment to the existing issue instead of opening a new one.

When opening a new issue, use the **Bug Report** template and fill in as much detail as possible. A good bug report should not leave others needing to chase you for more information.

### Suggesting Features

Feature requests are welcome! Use the **Feature Request** template and describe:

- The problem you're trying to solve — not just the solution you have in mind
- Why this would be useful to most users of the project
- Any alternatives you've considered

### Improving Documentation

Documentation is just as important as code. Typo fixes, clearer wording, missing examples — all of it is welcome and a great way to make your first contribution.

### Submitting Changes

1. **Fork** the repository and create your branch from `main`
2. **Make your changes**, following the conventions you see in the existing code
3. **Test** your changes — make sure nothing breaks and that any existing checks pass
4. **Open a Pull Request** and fill in the template

A few things to keep in mind:

- Keep your changes focused. One pull request should solve one problem — smaller PRs are reviewed faster and merged sooner.
- If your change is significant, consider opening an issue first to discuss the approach before investing time in it.
- Pull requests are **squash merged**, so don't worry about keeping a perfectly clean commit history on your branch — but do write a clear PR title, as it will become the commit message.
- Be patient and responsive during review. Reviews are conversations, not obstacles.

## Style Guidelines

- Follow the existing code style of the project. If the repository contains linter or formatter configuration files, use them.
- Write clear, descriptive commit messages and PR titles (e.g. `fix: handle empty input in parser` instead of `fixed stuff`).
- Keep documentation up to date when your change affects behavior.
